Etapa I: Funciones y Módulos

El misterio de los asteriscos: *args genera una tupla para argumentos posicionales, mientras que **kwargs construye un diccionario para argumentos de clave-valor.

Importaciones: Usar import modulo previene colisiones de nombres, y from modulo import funcion as f crea alias rápidos.

Estrategia de Importación

Sintaxis Escenario Real Ventaja Anti-Colisión
import modulo Recomendado por defecto. Usa modulo.funcion(); Python sabe exactamente de dónde viene.
from modulo import f Pocas funciones específicas. Llamada directa, pero riesgo si tienes funciones con nombre igual.
from modulo import f as alias Nombre largo o ya existente. El apodo único elimina cualquier conflicto de nombre original.
import modulo as alias Análisis de datos (ej. import pandas as pd). Conciso pero mantiene claridad con la notación de punto.
from modulo import * No recomendado profesionalmente. Peligro: Puede sobrescribir funciones locales sin aviso.

Implementación Técnica

def crear_perfil_usuario(nombre, apellido, **kwargs):
    # El diccionario kwargs captura datos adicionales ilimitados
    perfil = {'nombre': nombre, 'apellido': apellido}
    perfil.update(kwargs)
    return perfil

# Suposición de uso:
from perfiles_app import crear_perfil_usuario as nuevo_perfil

Etapa II: Safari de Objetos (POO)

1. ¿Qué función cumple __init__ al "nacer" un objeto y por qué self va primero?

El método __init__ inicializa los atributos. Mientras __new__ crea el objeto vacío, __init__ le asigna valores. Se usa self porque hace referencia al objeto específico creado, permitiendo que los valores se asignen a esa instancia y no a la clase en general.

2. ¿Qué hace la línea "mágica" super().__init__()?

Extiende las funcionalidades de la clase padre a la clase hija. Si se omite, el programa falla porque las propiedades del padre no existirían en la instancia del hijo, dejando al objeto "incompleto" respecto a su herencia.

3. ¿Cuándo elegir Composición sobre Herencia? (Caso Universidad)

Se elige cuando la herencia genera conflictos de roles. Ejemplo: En una universidad, si un Profesor también estudia, la herencia de Persona en ambos entra en conflicto. Es mejor que la Persona esté compuesta por roles; así, una persona puede tener el rol de alumno y profesor simultáneamente sin problemas estructurales.

Etapa III: Simulacro de Desastre

1. ¿Qué ventaja crítica ofrece with open() sobre el cierre manual?

La declaración with garantiza que el archivo se cierre automáticamente en cuanto termina el bloque de código, incluso si ocurre un error inesperado. El cierre manual con .close() es arriesgado, ya que un olvido o un fallo previo en el script dejaría el archivo abierto, consumiendo memoria o corrompiendo datos.

2. ¿Por qué no poner todo el código dentro de un try?

Es una mala práctica porque dificulta identificar qué línea específica causó el error. El bloque try debe contener únicamente la operación con potencial de fallo. En el bloque else, se deben alojar las instrucciones que dependen del éxito del try, manteniendo la lógica separada y limpia.

3. ¿Por qué preferir json.dump/load sobre texto plano?

El módulo JSON permite guardar estructuras complejas (listas, diccionarios) de forma nativa. A diferencia del texto plano (.write), que guarda todo como simples letras, JSON es un formato ligero, estructurado y portátil que permite que los datos sean leídos por casi cualquier lenguaje de programación sin perder su forma original.

Protocolo de Recuperación de Datos

import json

archivo_nombre = "configuracion_usuarios.json"

try:
    # Solo la operación crítica va dentro del try
    with open(archivo_nombre, "r") as archivo:
        configuracion = json.load(archivo)
except FileNotFoundError:
    # Manejo de fallo silencioso o por defecto
    configuracion = {"tema": "claro", "notificaciones": True}
    pass
else:
    print("✅ Configuración cargada con éxito.")

Etapa V: El Arquitecto Web

Mini-Ensayo: La POO como Cimiento de Django

Entender la Programación Orientada a Objetos (POO) no es solo un capricho técnico, sino la columna vertebral absoluta para construir arquitecturas web con Django. El framework funciona bajo un paradigma donde la base de datos no se manipula con código SQL en bruto, sino a través de un potente ORM (Object-Relational Mapping).

El Puente Lógico: Al crear un modelo en models.py, diseñamos una "Clase hija" que hereda de models.Model. Gracias a la POO, esta clase se transforma estructuralmente en una tabla entera, y sus atributos en las columnas de la base de datos.

Sin el dominio de las Clases, la Herencia y la Instanciación (Etapa 2), sería imposible comprender cómo Django interconecta la información, como el uso de ForeignKey para relacionar una Entrada con un Tema. Entender el "por qué" detrás del molde nos da el poder para modelar sistemas complejos escribiendo código Python puro, ordenado y escalable.

Conceptos Clave de Integración:

  • {{ variable }}: El lenguaje de etiquetas para imprimir datos en el Frontend.
  • {% logic %}: Control de flujo (bucles y condiciones) dentro de HTML.
